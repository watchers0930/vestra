# 2단계 검증 원칙 (Two-Stage Principle)

## Pattern

VESTRA의 핵심 설계 철학은 **중요한 판단이나 상태 변경을 단일 단계로 처리하지 않는다**는 것이다. 알고리즘 수준에서도, 배포 절차에서도, 보안 방어에서도 동일한 패턴이 반복된다: 1차 빠른 판단 → 2차 정밀 확인. 1차가 아무리 정확해도 2차를 생략하지 않는다.

이 패턴이 여러 도메인에서 독립적으로 같은 형태로 수렴했다는 점이 흥미롭다. 이것은 단순한 코딩 관례가 아니라, 불가역적 결과(운영 배포, 등기사고 미감지, 비용 폭발)를 막기 위한 시스템 설계 원칙이다.

## Instances

- **알고리즘 — V-Score**: 기본 V-Score 산출(1단계) → 교차분석 가중 조정(2단계). 단순 점수 계산에서 끝내지 않고, 감정가·실거래가·공시가 삼각검증으로 한 번 더 보정한다.
- **알고리즘 — 등기변동 하이브리드 감시**: Tilko 프리체크(1단계, 비용 절감) → CODEF 확정 조회(2단계, 신뢰도 확보). 1단계가 "변동 없음"으로 판정하면 2단계는 생략(비용 최적화), "의심"이면 2단계를 무조건 실행한다.
- **배포 — 2단계 배포 모델**: `npm run deploy:preview` → `t-vestra.vercel.app` 검증(1단계) → `npm run deploy:promote` 운영 승격(2단계). 동일 artifact를 그대로 승격하여 "검증된 것"만 운영에 배포한다.
- **보안 — 이중 방어**: Rate Limit(분당 30회, 1단계 빈도 제한) → Cost Guard(OpenAI 일일 비용 상한, 2단계 비용 차단). 둘 중 하나가 fail-open이 되어도 다른 하나가 동작한다.

## What This Means

단일 판단에 의존하는 시스템은 오탐(false positive)이나 비용 폭발 같은 한 가지 실패 모드에 취약하다. VESTRA가 2단계를 유지하는 것은 **1단계를 신뢰하지 않아서가 아니라, 결과가 불가역적이기 때문이다**.

등기감시에서 1단계를 빠르게 처리하는 이유가 비용 절감이듯, 이 원칙은 속도와 신뢰도의 트레이드오프를 명시적으로 분리한다. 새 기능 설계 시 "단일 단계로 끝나는 판단인가, 2단계 구조가 필요한가?"를 먼저 물어봐야 한다.
