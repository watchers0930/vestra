1. 플랫폼 개요

VESTRA는 LLM(Large Language Model) 기반 통합 AI 부동산 자산관리 플랫폼으로, 전세 안전성 분석, 등기부등본 권리분석, 계약서 위험도 평가, 세금 계산, 시세 전망, 사업성 분석(SCR) 등 부동산 의사결정에 필요한 전 영역을 AI로 자동화합니다.

1.1 시스템 아키텍처

Architecture Pattern
Next.js 16 App Router 기반 풀스택 아키텍처. 서버 컴포넌트(RSC)로 초기 렌더링 최적화, API Route Handler로 서버사이드 AI 연산 처리, PostgreSQL(Neon)로 영속 데이터 관리.

클라이언트 요청은 Vercel Edge Network를 통해 수신되며, 미들웨어에서 인증 상태를 확인한 후 적절한 핸들러로 라우팅됩니다. AI 분석 요청은 서버사이드에서 처리되어 API 키 노출 없이 OpenAI GPT-4o 및 자체 분석 엔진을 호출합니다.

2. 인공지능 및 데이터 분석

VESTRA의 AI 엔진은 외부 ML 라이브러리에 의존하지 않고, 순수 TypeScript로 구현된 자체 알고리즘을 사용합니다. 이는 모델의 판단 근거를 투명하게 제공하기 위한 설계 결정으로, 모든 분석 결과에 대해 요인별 기여도와 판단 사유를 설명할 수 있습니다.

  • 12개월 적중률(실제값이 신뢰구간 내 포함된 비율) 산출

3. 개인정보 보호

3.1 PII 암호화 (AES-256-GCM)

lib/crypto.ts

개인식별정보(PII)는 NIST 승인 인증 암호화 알고리즘인 AES-256-GCM으로 암호화하여 저장합니다. 이는 블록체인 기술과 별개인 범용 대칭키 암호화 표준으로, 데이터 기밀성과 무결성을 동시에 보장합니다.

항목사양
암호화 알고리즘AES-256-GCM (Galois/Counter Mode)
키 파생SCRYPT (N=16384, r=8, p=1)
키 입력AUTH_SECRET + PII_SALT → 32바이트 키
초기화 벡터16바이트 랜덤 IV (매 암호화마다 새로 생성)
인증 태그16바이트 (변조 탐지)
저장 형식Base64(IV + AuthTag + Ciphertext)
검색 지원HMAC-SHA256 해시 인덱스 (암호화 필드 조회용)
동일 평문 → 다른 암호문
매 암호화마다 랜덤 IV를 생성하므로, 동일한 개인정보를 암호화해도 매번 다른 암호문이 생성됩니다. 이는 패턴 분석을 통한 평문 추론 공격을 원천 차단합니다.

4. 블록체인 암호화 기법 적용

VESTRA는 분산 원장(Distributed Ledger)이 아닌, 블록체인의 핵심 암호화 기법을 부동산 등기 데이터의 무결성 검증에 선별적으로 적용합니다. 비트코인의 머클 트리(Merkle Tree)와 블록 해시 체인(Hash Chain) 구조를 차용하여, 분석 파이프라인과 등기 변동 이력의 위변조를 원천 탐지합니다.

블록체인 기법 vs. 블록체인
본 플랫폼은 분산 합의(Consensus), 채굴(Mining), 토큰 경제를 사용하지 않습니다. 블록체인의 암호학적 자료구조(머클 트리, 해시 체인, 디지털 서명)만을 활용하여, 중앙 서버 환경에서도 데이터 무결성을 수학적으로 증명할 수 있는 구조를 구현합니다.

4.1 머클 트리 무결성 검증

lib/integrity-chain.ts 블록체인 핵심 자료구조

비트코인이 트랜잭션 검증에 사용하는 것과 동일한 머클 트리(Merkle Tree) 구조를 적용합니다. 분석 파이프라인의 각 단계(파싱 → 위험도 산정 → 검증)를 SHA-256 해시 트리로 연결하여, 어느 단계에서든 데이터 변조가 발생하면 루트 해시 불일치로 즉시 탐지합니다.

구현 구조

  • 리프 해시 — SHA-256('leaf:' + data) — 각 분석 단계의 입출력 데이터
  • 노드 해시 — SHA-256('node:' + leftHash + ':' + rightHash) — 상위 노드 결합
  • 균형 트리 — 홀수 리프 시 마지막 노드 복제 (비트코인과 동일 방식)
  • 머클 증명(Merkle Proof) — 리프 인덱스 + 형제 경로로 루트까지 검증. 전체 데이터 없이도 특정 항목의 포함 여부를 O(log n)으로 증명
Leaf₁ → SHA-256 ─┐
                              ├→ Node₁₂ ─┐
Leaf₂ → SHA-256 ─┘                       ├→ Merkle Root (불변 증거)
Leaf₃ → SHA-256 ─┐                       │
                              ├→ Node₃₄ ─┘
Leaf₄ → SHA-256 ─┘

4.2 등기 스냅샷 해시 체인

lib/registry-blockchain.ts 블록체인 블록 구조

블록체인에서 각 블록이 이전 블록의 해시를 포함하는 것과 동일한 원리로, 등기부등본의 시점별 스냅샷을 해시 체인으로 연결합니다. 체인의 어느 한 지점이라도 변조되면 이후 모든 해시가 불일치하여 탐지됩니다.

필드타입블록체인 대응 개념설명
sequenceNoIntBlock Height스냅샷 순번 (시간순 정렬)
merkleRootStringTx Merkle Root섹션별 해시의 머클 트리 루트
snapshotHashStringBlock Hash현재 스냅샷 전체 SHA-256 해시
previousSnapshotHashStringPrevious Block Hash이전 스냅샷 해시 (체인 연결)
signatureStringBlock SignatureEd25519 디지털 서명 (진위 검증)
encryptedDataString-AES-256-GCM 암호화된 등기 전문
sectionHashesJSONTx Hashes갑구/을구/표제부 개별 해시
변동 탐지 프로세스
① 등기부등본 원문을 섹션(표제부/갑구/을구)별로 분리 → ② 각 섹션을 SHA-256으로 해싱 → ③ 섹션 해시들로 머클 트리 구성 → ④ 이전 스냅샷 해시 + 현재 머클 루트로 체인 해시 생성 → ⑤ Ed25519 서명으로 진위 봉인

이후 검증 시, 체인의 임의 지점에서 해시를 재계산하여 저장된 값과 비교합니다. 불일치 발생 시 정확히 어느 섹션이 변경되었는지까지 식별 가능합니다.

5.2 권리 그래프 분석 엔진

lib/rights-graph-engine.ts

등기부등본의 권리관계를 방향성 그래프(Directed Graph)로 모델링하여 구조적 위험을 분석합니다. 부동산을 루트 노드로, 각 권리(소유권, 근저당, 압류 등)를 노드로, 권리 간 관계를 엣지로 표현합니다.

분석 기능

  • 그래프 구축 — 갑구/을구 항목을 노드와 엣지로 변환, 말소된 항목 자동 제외
  • 순환 탐지 — DFS 기반 사이클 감지 (권리 간 순환 참조 경고)
  • 체인 분석 — 루트에서 말단까지 경로 추적, 권리 체인 식별
  • 위험 전파 — 상위 권리의 위험이 하위 권리로 전파되는 시뮬레이션
  • 크리티컬 패스 — 최대 손실 경로 식별 및 최대 손실 금액 산출
